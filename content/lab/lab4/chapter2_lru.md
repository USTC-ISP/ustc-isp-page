## 1. 引入

在计算机系统的设计中，**缓存（Cache）** 是一种用于提高数据访问速度的技术。然而，缓存的物理空间往往是有限的．当缓存存满时，若有新的数据需要存入，就必须从已有的数据中选出一个“牺牲品”将其移出，为新内容腾出位置。

这一过程被称为 **缓存淘汰（Cache Replacement）**。为了使缓存的命中率尽可能高，人们设计了多种调度算法，其中最为经典且应用最广的便是 **LRU (Least Recently Used)** 算法．

## 2. LRU 的基本原理

LRU，即“最近最少使用”，其核心思想基于**时间局部性理论**：**“如果一段数据最近被访问过，那么它在将来被访问的概率也会更高；反之，长期未被访问的数据，在将来被访问的概率较低，应当优先被淘汰．”**

### 2.1 算法模型

LRU 通常可以抽象为一个按“访问时间”排序的队列：
1.  **最新鲜的数据**（最近被访问过）放在队列的头部。
2.  **最陈旧的数据**（最久没被访问过）放在队列的尾部。

### 2.2 核心操作流程

假设缓存容量为 \(k\)：

-   **访问数据 (Get)**：
    -   如果在缓存中找到了对应的 `key`，则将该数据挪到队列的最前端，表示它刚被“刷新”过。
    -   如果没找到，返回失败标识。
-   **更新/添加数据 (Put)**：
    -   如果数据已存在，更新其 `value` 并移动到队列最前端。
    -   如果数据不存在：
        -   若队列已满，则将**队列尾部**（最陈旧）的数据剔除并释放。
        -   将新数据插入到队列**头部**。

## 3. 运行示例

假设缓存容量为 \(3\)，访问序列为：`put(1,1), put(2,2), put(3,3), get(1), put(4,4)`．

1.  `put(1,1)`：缓存为 `[1]`．
2.  `put(2,2)`：缓存为 `[2, 1]`．（新数据插在头部）
3.  `put(3,3)`：缓存为 `[3, 2, 1]`．
4.  `get(1)`：命中了 `1`，将其移动到头部，缓存变为 `[1, 3, 2]`．
5.  `put(4,4)`：缓存已满，淘汰末尾的 `2`，插入 `4` 到头部，缓存变为 `[4, 1, 3]`．

## 4. LRU 的典型应用

LRU 并不是只存在于教科书中的理论，它广泛应用于工业界系统的各个角落：
-   **操作系统**：页表置换（Page Replacement），内核会维护一个内存页链表，淘汰长期不活动的物理页到磁盘。
-   **数据库**：如 Redis 或 MySQL 的 Buffer Pool，用于决定哪些索引或行数据保留在内存中。
-   **Web 服务器**：如 Nginx 或 CDN 节点，通过 LRU 决定哪些高频访问的文件存放在内存缓存中以降低响应延迟。

## 5. 其他常见的替换算法

除了 LRU，还有一些针对不同场景设计的算法：
-   **FIFO (First-In, First-Out)**：先进先出。最简单的策略，不考虑访问频率，可能导致性能较差（Belady 异常）。
-   **LFU (Least Frequently Used)**：最不经常使用。记录每个数据被访问的**次数**，淘汰访问次数最少的。适用于某些长期高频访问的数据，但难以应对“瞬间高频随后冷寂”的情况。
-   **Clock (时钟算法)**：LRU 的一种近似实现。它像钟表盘一样循环检查数据的使用位（Referenced Bit），硬件开销更小。
-   **ARC (Adaptive Replacement Cache)**：自适应缓存。结合了 LRU 和 LFU 的优点，能根据工作负载实时调整策略，是高性能存储系统中常用的算法。

## 6. 本实验的实现建议

### 6.1 为什么要移动节点？
为了方便管理，我们约定：
-   **链表头节点之后的第一个节点**始终是“最近访问”的节点。
-   **链表的最后一个节点**（尾节点）始终是“最久未访问”的节点。

因此，每次执行 `get` 或 `put` 命中已有数据时，你需要执行“摘除节点”并“头插至首位”的操作。

### 6.2 进阶思考
虽然单链表实现简单，但查找数据节点的时间复杂度是 $O(n)$。在生产环境中，LRU 通常要求 $O(1)$ 的时间复杂度。你可以思考：
-   如果要实现 $O(1)$ 的快速定位和移动，通常需要组合使用什么样的数据结构？
